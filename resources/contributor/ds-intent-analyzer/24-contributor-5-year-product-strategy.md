# `ds-intent-analyzer` 5-Year Product Strategy

## Purpose

This document defines the internal 5-year product strategy for `ds-intent-analyzer`.

It is written for:

- maintainers
- PM-style product owners
- contributors making roadmap decisions

It is not intended as external marketing copy.

The goal is to translate:

- current product doctrine
- external market developments
- repo-native validation evidence

into a clear long-horizon direction with measurable objectives.

---

## 1. 2031 Vision Statement

By **2031**, `ds-intent-analyzer` should be the **lightweight decision layer for agentic design-system work**.

It should sit between:

- design context
- repo evidence
- coding agents

and help teams turn UI discomfort, design drift, or reference-system curiosity into:

- grounded diagnosis
- disciplined borrowing logic
- audit-quality system direction
- executable next actions

The product should win on:

- evidence-grounded judgment
- audit quality
- borrowing discipline
- execution-ready outputs

The product should not drift into:

- generic component generation
- blind reference cloning
- hidden orchestration
- hidden memory behavior
- becoming a design platform

---

## 2. Product Position and Who We Serve

### Position

`ds-intent-analyzer` is not a generic coding assistant and it is not a visual design generator.

Its strongest product position is:

- a Codex-first design-system judgment layer
- for agents and teams working from real repo and UI evidence
- with explicit outputs another agent can actually use

### Primary users

- Codex and Codex-like coding agents
- engineers and designer-developers working from real repos
- maintainers who need repeatable DS review and audit outputs

### Core user jobs

- explain why the current UI feels off
- audit a current DS direction without falling into memo-like theory
- borrow from a reference system without cloning it
- produce one bounded next action another agent can execute
- reuse recurring review outputs across time without hidden state

### Repo-grounded proof surfaces

Current proof in this repo already supports this position:

- standing maintainer gate for recurring usefulness
- client-repo dogfood in clean downstream repos
- explicit `Audit handoff` and reusable `audit-packet.md`
- recurring review artifacts
- lean private distribution across local path, git source, and tarball installs

This means the next 5 years should optimize for **better evidence, better handoffs, and safer team usage**, not for broader but shallower feature sprawl.

## 3. Strategic Pillars

### Pillar 1 — Evidence-native reasoning

The product should get stronger as evidence quality improves.

Winning direction:

- richer current-state inputs
- stronger use of screenshots and repo summaries
- tighter use of `audit-evidence.md`
- future design-context connectors when they clear the lightweight-first bar

Why:

- the strongest market trend is toward evidence-rich agent workflows
- the strongest current product ceiling is still evidence quality, not vocabulary breadth

### Pillar 2 — Execution-ready outputs

The product should end in action, not just analysis.

Winning direction:

- stronger `Audit handoff`
- reusable `audit-packet.md`
- clearer recurring review artifacts
- better bounded coordination packets for downstream agents

Why:

- handoff quality remains a large unresolved problem in the market
- repo-native validation already shows that bounded handoffs are one of the product's strongest differentiators

### Pillar 3 — Governed team adoption

The product should stay easy to trust inside real teams.

Winning direction:

- private team-sharing workflows
- lean package surface
- explicit writes only
- auditable behavior
- update/install paths that are safe and predictable

Why:

- governance, sandboxing, and package hygiene are increasingly part of the product bar for agent tools
- in this category, operational trust is product value

### Pillar 4 — Lightweight transparent operation

The product should remain small, explicit, and legible.

Winning direction:

- Codex/GPT-first behavior
- explicit artifacts
- thin helpers only when justified
- no hidden framework behavior
- no magic memory layer

Why:

- this is the repo doctrine
- it is also the best defense against feature sprawl and trust erosion over 5 years

## 4. 5-Year Horizon Plan

### Phase 1 — 2026 to 2027

Theme:

- strengthen evidence quality and artifact usefulness

Primary outcomes:

- improve install-to-first-value in clean client repos
- make structured evidence more common in real audit flows
- make artifact outputs easier to accept without rewrite
- keep recurring review compact and useful

Typical product bets:

- richer evidence intake conventions
- stronger audit/review artifact quality
- continued downstream proof across supported install channels

### Phase 2 — 2028 to 2029

Theme:

- make recurring review and structured evidence durable team workflows

Primary outcomes:

- turn recurring review from a validated workflow into a dependable team habit
- deepen safe private sharing across teams and repos
- reduce the number of rich-evidence audits that still behave like thin brief-only reads

Typical product bets:

- stronger evidence-backed audits
- sharper drift detection in review cycles
- better internal rollout and adoption support for private team usage

### Phase 3 — 2030 to 2031

Theme:

- become the standard internal decision layer for DS audit/review work

Primary outcomes:

- multi-repo or multi-team usage becomes normal
- the product is trusted as the first stop for DS audit and handoff direction
- recurring review outputs are reused over time instead of recreated from scratch

Typical product bets:

- mature internal adoption patterns
- durable review workflows
- high-trust repo-safe operation at team scale

## 5. Measurable Objectives and KPIs

Use two KPI layers:

- **Primary leading KPIs** for product health that the repo can already measure or approximate
- **Secondary adoption KPIs** for later pilot or team telemetry

### Primary leading KPIs

#### A. Artifact acceptance rate

Definition:

- the share of qualifying sessions where one of these outputs is accepted without major rewrite:
  - `Audit handoff`
  - `audit-packet.md`
  - `review-log.md`
  - `review-brief.md`
  - `Frontend handoff`

Why this matters:

- this is the clearest measure that the product turns judgment into execution

#### B. Evidence-rich audit rate

Definition:

- the share of audits using structured evidence beyond prompt-only or brief-only input

Why this matters:

- it measures whether the product is actually becoming evidence-native

#### C. Recurring review usefulness rate

Definition:

- the share of repeated review runs that identify meaningful drift or changed priority rather than repeating the first-run memo

Why this matters:

- it measures whether recurring review is becoming a real workflow surface instead of a renamed one-shot audit

#### D. Supported install success rate

Definition:

- success rate across validated downstream install and update channels

Why this matters:

- governed adoption depends on downstream reliability

#### E. Explicit-write safety rate

Definition:

- keep silent writes, hidden memory behavior, and package-owned scheduler behavior at zero

Why this matters:

- trust erosion here would directly damage the product identity

### Secondary adoption KPIs

Use these only when pilot or team telemetry becomes available:

- active teams using the skill in recurring workflows
- active client repos using the installed skill
- recurring artifact reuse across review cycles
- install-to-first-value time in real team environments

### Objective targets

#### By end of 2027

- `>= 90%` pass rate across the standing maintainer gate, external dogfood, and distribution proof surfaces
- `>= 5` first-class evidence input types supported in product guidance
- `>= 50%` artifact acceptance rate in qualifying validation or dogfood scenarios
- `100%` explicit-write safety rate

#### By end of 2029

- `50%` reduction in rich-evidence audits that still behave like thin `brief only` cases, measured against a 2026 baseline
- `>= 70%` recurring review usefulness rate in repeated synthetic review cycles
- `>= 75%` artifact acceptance rate in qualifying external dogfood or pilot-style sessions
- `>= 95%` supported install success rate

#### By end of 2031

- `>= 80%` artifact acceptance rate
- `>= 80%` recurring review usefulness rate
- `0` hidden writes or hidden memory behavior
- cautious long-horizon adoption goal:
  - `20 active teams` or `50 active client repos`

Important note:

- the 2031 adoption target is a long-horizon directional objective, not a near-term commitment
- it should not override product-health KPIs while telemetry is still limited

## 6. Risks and Anti-Goals

### Main risks

- drifting into generic AI design or component generation instead of decision quality
- overbuilding automation before evidence-rich usefulness is strong enough
- turning helpful artifacts into hidden framework behavior
- shipping more retrieval or memory complexity than the product bar can justify
- optimizing for source-repo dogfood convenience instead of client-repo value

### Anti-goals

Do not optimize this product toward:

- becoming a Figma replacement or design platform
- blind cloning of popular reference systems
- hidden orchestration or autonomous scheduler behavior by default
- opaque memory accumulation
- taxonomy complexity that outgrows user-visible value

### Doctrine alignment

This strategy depends on preserving the current product doctrine from:

- `09-contributor-product-doctrine.md`

It also depends on continuing to use the market read from:

- `23-contributor-market-monitoring.md`

as an evidence filter rather than as an excuse for trend-driven feature sprawl.
